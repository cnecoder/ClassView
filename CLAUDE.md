# 必须遵守的规则

减少LLM常见编码错误的行为指南。按需与项目特定说明合并使用。
权衡：本指南偏向谨慎而非速度。对于简单任务，可自行判断。

## 规则——编码前先读先思考
不假设。不隐藏困惑。暴露权衡。
实施前：
- 编码之前先读取相关部分，理解意图
- 明确陈述你的假设。不确定时先问。
- 存在多种理解方式时，列出它们——不要擅自选择。
- 存在更简单的方案时，指出来。必要时提出反对。
- 遇到不清楚的地方，停下来。明确指出哪里不清楚，然后提问。

## 规则——简洁优先
用最少代码解决问题。不写推测性代码。
- 不添加超出需求的功能。
- 不为单次使用的代码创建抽象。
- 不添加未被要求的「灵活性」或「可配置性」。
- 不为不可能发生的场景添加错误处理。
- 如果你写了 200 行但 50 行就够，重写它。
- 自问：「资深工程师看了会说这太复杂吗？」如果会，就简化。

## 规则——精准修改
只动必须改的。只清理自己造成的混乱。
编辑已有代码时：
- 不要「改进」相邻的代码、注释或格式。
- 不重构没有坏的东西。
- 匹配现有风格，当有更好的实现，倾向于不同写法时明确提出，同时给出更好的原因，交给用户选择。
- 如果你注意到无关的死代码，提出来——不要擅自删除。
当你的改动产生死代码时：
- 删除因你的改动而不再使用的 import / 变量 / 函数。
- 不要删除原有死代码，除非被要求。
- 检验标准：每一个被修改的行都应能直接追溯到用户的需求。

## 规则——目标驱动执行
定义成功标准。循环直到验证通过。
将任务转化为可验证的目标：
- 「添加校验」→「先为非法输入编写测试，再让测试通过」
- 「修复这个 bug」→「先写能复现该 bug 的测试，再让测试通过」
- 「重构 X」→「确保重构前后测试均通过」
对于多步骤任务，先陈述简要计划：
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]

强有力的成功标准让你能够独立迭代。弱标准（「让它跑起来」）则需要不断的澄清确认。

## 规则——暴露失败
如果任何事被悄悄跳过，"已完成"就是错的。
如果任何测试被跳过，"测试通过"就是错的。
默认暴露不确定性，不要隐藏。

## 规则——语言使用规范
编写文档、注释或任何书面内容时，主要语言必须为中文。但以下例外情况必须保留原始非中文形式（如英文、数字、符号、技术术语、代码与伪代码、命令行指令、引用的原始术语或缩写、固定标识符），不得翻译或汉化。
违反此规则可能导致误解或技术沟通问题。请严格遵守。

## 规则——Git Commit 规范
- 提交信息遵循 `.gitcommit-template` 模板格式
- 提交前确认所有修改文件已 `git add`
- 使用 `git commit -m "$(cat <<'EOF' ... EOF)"` 格式提交多行消息
